Undoing Commits

Tower offers you two different ways to undo the changes introduced by certain commits:


Reverting commits

Reverting will produce a new commit that effectively is the opposite of the commit to revert. Git will apply changes that reverse the effect of that commit's changes. In your repository's history, both commits - the original and the reverting one - will be visible.

To revert a commit

  • Select the commit (or even multiple commits) that you want to revert in the "History List" view.

  • Right-click the commit and select "Revert This Commit..." from the contextual menu.

  • In the following dialog, select "Automatically Commit Reverted Changes" if you want Tower to directly commit the reversing changes. If you don't select this option the changes reversing the commit will be applied to your working copy, leaving it up to you to further modify, stage, and commit them.

Resetting the current HEAD

Another way to undo commits is to reset HEAD to a previous commit. Your current branch will then point at the specified older commit, effectively removing the newer commits from your history.

To reset the current HEAD to a previous commit

  • Select the commit you want your current HEAD branch to point to.

  • Right-click the commit and select "Reset HEAD to This Commit..." from the contextual menu.

  • In the following dialog, you can choose how Git should treat changes:

    • "Discard changes" (»hard« reset)
      All current changes in your working copy (staged or unstaged) and all committed changes since the chosen commit will be discarded. You will have a clean working directory and an empty staging area. NOTE:Uncommitted changes will be lost and cannot be restored.

    • "Preserve all changes" (»mixed« reset)
      All changes - including uncommitted local ones - are preserved: committed changes since the chosen commit will be applied as unstaged changes to your working directory, while current local changes will be preserved, too. Your staging area will be empty.

    • "Preserve new local changes" (»keep« reset)
      Committed changes since the chosen commit will be discarded, while uncommitted local changes in your working tree are preserved. The reset will be aborted if there could be conflicts between the changes in the commits to be removed and the current changes in the working directory that shall be preserved.
